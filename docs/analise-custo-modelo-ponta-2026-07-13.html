Vale a pena rodar um modelo de ponta pra montar treino?

Resposta curta: sim — e por menos do que você paga hoje. Mas não trocando o modelo. Trocando o prompt primeiro. Do jeito que o build está montado agora, um modelo de ponta multiplica uma conta que já está no vermelho.

O que eu não esperava encontrar. O build de treino já é deficitário hoje, com o modelo “barato”. Um build real de 13/jul consumiu 320 mil tokens de input e custou $0,51 (R$ 2,81). Ele é cobrado como 6 créditos — ou seja, R$ 0,47 de custo por crédito. No plano Pro IA você vende crédito a R$ 0,27, e no Premium a R$ 0,13.

Cada treino montado por um assinante Pro ou Premium dá prejuízo hoje. Antes de discutir GPT-5.5, é isso que precisa ser consertado — e a boa notícia é que o conserto é o mesmo que torna o modelo de ponta acessível.

02Por que 320 mil tokens

Não é o treino que é grande — a saída tem 3,4 mil tokens. O input infla por três motivos que se multiplicam:

  1. O catálogo inteiro de tools vai em todo passo. São 62 tools ≈ 16 mil tokens de schema. Num turno de 10 passos, isso é 160 mil tokens só de definição de ferramenta — inclusive as de financeiro, agenda e leads, num turno que só monta treino. O subsetting por intenção existia e foi removido hoje (command-engine.ts:606-614), com a justificativa de que “o catálogo completo é barato”. A medição não sustenta isso: ele é o custo.
  2. O loop reenvia tudo a cada passo. Teto de 16 passos, 8–20 tool calls. Os 320k não são uma janela de 320k — são a soma de ~10 passos de ~30k, com o mesmo prefixo cobrado de novo em cada um.
  3. O cache não é aproveitado — e, pior, não é medido. Não existe cache_control, cachedContent nem leitura de cachedInputTokens em lugar nenhum do repo: o metering (command-engine.ts:1112) grava só input e output. Por isso cached_input_tokens é nulo em 100% dos eventos. Isso não prova que o cache nunca acerta — prova que você está cego para ele. O $0,51 registrado assume zero cache: é o teto, e o custo real está em algum ponto entre os painéis A e B do gráfico abaixo.

Os três se resolvem sem trocar de modelo. É exatamente isso que torna a pergunta do modelo respondível.

04Quanto custaria cada modelo

Todos os painéis usam o mesmo turno de build medido em produção — 320k de input, 3,4k de saída. Só a arquitetura do prompt muda.

A leitura que importa: o GPT-5.5 no painel C ($0,19) é 2,6× mais barato que o Gemini Flash no painel A ($0,51) — que é a sua situação de hoje. Dá para subir para o melhor modelo do mercado e gastar menos. O que não dá é subir de modelo sem mexer no prompt: aí o mesmo GPT-5.5 custa $1,70 e a conta triplica.

Três conclusões saem daqui, e nenhuma é sobre o preço dos modelos:

  • O prejuízo já existe. Um Pro IA que usar a cota inteira montando treinos (50 builds) gera R$ 79,90 de receita e R$ 140 de custo. O Premium é pior. Isso não aparece no extrato porque a base ativa é minúscula — 26 treinadores, 8 assinaturas, todas no Essencial, e apenas 3 builds em Gemini em um mês. É um prejuízo que só se materializa quando o produto der certo.
  • Corrigido o prompt, o modelo de ponta cabe — em Essencial e Pro. GPT-5.5 e Opus 4.8 pós-conserto ficam com margem positiva nos dois planos que respondem por toda a receita atual.
  • O Premium IA está mal precificado, não importa o modelo. A R$ 0,13/crédito, só o gpt-4.1-mini e o Sonnet 5 fecham. Se você quer GPT-5.5 no Premium, o caminho é subir o peso do build de 6 para ~9 créditos — que, aliás, já era a recomendação nº 1 do doc de 07/jul.

06Recomendação

  1. Medir o cache antes de tudo. Ler cachedInputTokens do providerMetadata e gravar no metering (command-engine.ts:1112). É uma linha, e hoje você não sabe se paga $0,51 ou $0,25 por build. Nenhuma decisão de modelo deveria ser tomada às cegas sobre a variável que domina o custo.
  2. Ligar o caching de verdade. Prefixo estável (instruções + tools primeiro; contexto dinâmico depois — hoje o dinâmico fica no meio do system e pode estar quebrando o prefixo cacheável), cache_control explícito na Anthropic, contexto explícito no Gemini. Maior retorno por linha de código do projeto: corta ~67% do custo sem tocar em modelo nem em qualidade.
  3. Devolver o subsetting de tools. A remoção de hoje (“o catálogo completo é barato”) custa ~16k tokens por passo, ~160k por build. Não é barato — é a maior linha da conta.
  4. Só então trocar o modelo, e só no turno de build. Feito o de cima, o upgrade sai quase de graça. Eu iria de Claude Sonnet 5 como default de build — fecha margem nos três planos, custa menos que o Gemini de hoje e lidera tool-calling, que é literalmente o trabalho do build. GPT-5.5 ou Opus 4.8 como tier premium, atrás do gate de plano, se o head-to-head mostrar prescrição melhor.
  5. Deixar a conversa no gpt-4.1-mini. $0,007 por turno. Não há dinheiro nem qualidade a ganhar ali.

Se for pra fazer uma coisa só: ligue e meça o cache. Ele paga o modelo de ponta sozinho — e sem ele nenhum modelo, caro ou barato, fecha a conta do Pro e do Premium.

07Premissas e o que não verifiquei

  • 80% de taxa de cache (painéis B e C) é premissa, não medição — e é justamente o que a recomendação nº 1 existe para descobrir. Se o hit real for 60%, os números do painel C sobem ~30%; a conclusão não muda de sinal.
  • 65k de input no painel C é meta, não medição. Vem de cortar tools por intenção (16k → ~3k por passo) e filtrar o catálogo de exercícios. Não foi implementado nem medido.
  • Câmbio R$ 5,50/US$. As margens em reais escalam proporcionalmente.
  • Amostra pequena. Só existem 8 turnos de build em Gemini registrados em produção. Os $0,51 são de um build real e batem com outros três, mas não é uma distribuição.
  • Qualidade não foi medida. Esta análise responde “quanto custa”, não “prescreve melhor”. Que GPT-5.5 e Opus 4.8 montem treinos melhores que o Flash é hipótese plausível — e o jeito de resolver é um head-to-head nos evals que já existem em lib/assistant/evals/. Um modelo de ponta que erra menos e precisa de menos retry pode se pagar só nisso.
  • O portão LGPD continua aberto. Prescrição toca dado de saúde e não há nenhuma configuração de zero-retention/no-training em nenhum provider do repo. Trocar de modelo sem resolver isso troca a exposição de lugar, não a reduz.
