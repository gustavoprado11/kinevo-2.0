O treinador operando o Kinevo por IA — caminhos e oportunidades

Não é um chatbot, e não é IA para o aluno. É dar ao treinador uma forma de executar o trabalho dentro do Kinevo conversando, falando ou com um toque. A tese deste documento: como temos controle total do código, podemos fundir IA e produto de um jeito que o conector externo (Claude/ChatGPT) nunca alcança.

01 A reformulação

O MCP já prova que a IA sabe operar o Kinevo — 55 ações cobrindo alunos, prescrição, agenda, financeiro, avaliações, formulários e leads. Mas no Claude isso vive fora do produto: texto entra, texto sai, e o treinador alterna de app. Dentro do Kinevo, a IA pode deixar de ser uma "caixa de conversa" e virar uma camada de operação do próprio app.

Hoje · externo

Conector MCP

Claude/ChatGPT chamam as 55 tools por OAuth. Ótimo, mas é outro app, sem UI do Kinevo, sem estado compartilhado, sem proatividade.

Hoje · interno

Assistente in-app

api/assistant/chat com gpt-4.1-mini e só 3 tools. Tem streaming, contexto por aluno e hardening — mas é subutilizado.

A construir

Camada de operação

Uma engine única (as 55 tools via ponte in-memory) alimentando várias superfícies dentro do app: chat, barra de comando, UI generativa, agente proativo, voz.

02 O que ter o código desbloqueia (vs. conector externo)

Esta é a justificativa estratégica. Cada linha é algo impossível ou ruim no Claude externo e natural dentro do Kinevo:

CapacidadeNo conector externo (Claude)Dentro do Kinevo (código nosso)
UI generativa✗ só texto/markdown e JSON✓ devolve componentes vivos do Kinevo (tabela de treino editável, card financeiro, gráfico) via streamUI
Estado compartilhado✗ resultado fica no chat✓ a ação reflete na hora na GUI (otimista) — chat e tela são o mesmo estado
Embutido nas telas✗ é outro app✓ barra de comando (⌘K) e IA contextual em cada página (no perfil do aluno, na agenda…)
Proatividade✗ só reativo✓ usa o cron de insights e os drafts que já existem → briefing matinal, fila de aprovação
Voz & nativo~ via apps do Claude✓ voz no mobile + Siri/App Intents ("Ei Siri, marca o pagamento da Maria")
Confirmar/desfazer✗ sem undo, sem auditoria✓ card de confirmação (HITL) + feed "a IA fez X" com desfazer
Roteamento determinístico~ 20 tool-calls p/ prescrever✓ "gera programa" cai na action generateProgram (1 call, barato, validado)
Auth & custoOAuth + tokens✓ sessão Supabase (sem OAuth) — a mesma tool roda com o JWT do treinador

★ O grande diferencial: UI generativa + estado compartilhado

No Claude, "ajuste o treino da Marina" devolve texto descrevendo o ajuste. No Kinevo, a IA devolve a própria tabela de treino, editável, já refletida no programa — o treinador ajusta a carga ali e está salvo. A IA deixa de "falar sobre o app" e passa a renderizar o app. (AI SDK streamUI/createStreamableUI — hoje não usado, mas suportado.)

Copiloto de UI generativa

O diferencial

A IA devolve componentes vivos e editáveis do Kinevo — não texto. A conversa vira um canvas: tabela de treino, card de cobrança, calendário, ficha do aluno, tudo manipulável e já salvo no estado.

Reaproveita infra

🎙️

Mobile: voz + nativo (Siri)

Onde o treinador está

Na academia, sem teclado: ditar a intenção e confirmar com Face ID. E mais fundo: expor ações como App Intents/Siri Shortcuts — "Ei Siri, registra presença da Marina".

Esforço médio (voz) · alto (Siri)Diferencial mobile

Ref.: App Intents no Expo (config plugin) · ganho citado +8–12pt retenção D7
Fecha com o código: mobile chama www.kinevoapp.com/api/assistant/* com o JWT (padrão já usado). Siri = módulo nativo novo.

Como escolher

B, C e D dão o maior salto sobre o que o Claude faz porque dependem de termos o código. A recomendação (§06): construir a engine uma vez e acender as superfícies em ordem — começando pela barra de comando (B) + UI generativa (C), com o proativo (D) logo atrás reusando insights/drafts.

05 Como tudo fecha com o código

O ponto crítico que você pediu: as ideias só valem se fizerem sentido com o código. A boa notícia — todas compartilham uma engine, e o grosso dela já está especificado (F0 da spec). As superfícies são fachadas finas sobre ela.

Princípio de design: 1 engine, N fachadas

Não construir 5 produtos. Construir a engine uma vez (ponte + tool-policy + HITL + metering = o F0 já especificado) e then cada caminho é uma UI fina por cima. Isso mantém um catálogo só (o mesmo que serve o Claude externo continua intocado) e garante que tudo "fecha": cada superfície fala a mesma língua de tools, confirmação e créditos.

2 travas técnicas já investigadas (não são surpresa)

(1) O AI SDK 4.3.19 descarta as annotations das tools na ponte → a classificação read/write tem que ser estática em tool-policy.ts. (2) Toda tool vem com execute automático → o HITL exige substituir as tools de confirmação por versões client-side + endpoint execute-tool. Ambas já têm solução na spec.

06 Sequência recomendada

F0
engine

Construir a engine uma vez

Ponte in-memory + tool-policy.ts + execute-tool (HITL) + metering/cota + trainers.ai_tier. Nada visível — mas habilita TODAS as superfícies. (É o F0 da spec.)

1

Barra de comando ⌘K (caminho B)

Maior retorno/menor esforço: a IA opera a tela atual de qualquer lugar. Reaproveita o palette. Prova o valor sem nova tela.

2

UI generativa (caminho C)

O diferencial: respostas viram componentes vivos (treino editável, card de cobrança). É o que o Claude não faz. Pode nascer dentro do ⌘K e do chat.

3

Agente proativo (caminho D)

Briefing + fila de aprovação reusando insights/drafts. Vira hábito diário e retenção. Baixo custo incremental.

4

Mobile: chat contextual + voz (caminho E)

Mesma API via JWT. Voz (expo-speech/Whisper) + push→chat. Siri/App Intents como aposta seguinte (módulo nativo).

Onde o chat-first (A) entra

Não desaparece — vira a casa dessas superfícies na fase 2/3 ("chat como tela inicial", Premium). Mas começar por ⌘K + UI generativa prova valor mais rápido e com menos risco de "página em branco".

07 Riscos & decisões em aberto

1
Custo escala com uso. Power-user via chat ≈ R$180/mês de API. Metering por créditos + cap por plano precisa existir antes de abrir. (Já há tabela/RPC na spec.)
2
Confiança em escrita. Toda ação sensível passa pelo gate confirm (HITL) — nunca auto-executa cobrança/cancelamento. Considerar um feed "IA fez X" com desfazer para as não-sensíveis.
3
Quantas superfícies de uma vez? Recomendo 1 ponta-de-lança (⌘K) + UI generativa, não as 5. Evita dispersão antes de provar valor.
4
Tier não existe hoje (acesso é binário). Precisa trainers.ai_tier + stripe_price_id via webhook. Decisão de pricing (Essencial/Pro IA/Premium IA) antes da fase 1.
5
Siri/App Intents dá retenção mas custa módulo nativo. Vale como v2 do mobile, não no v1.
6
Modelo. gpt-4.1-mini hoje; UI generativa e seleção entre 55 tools podem pedir medir acurácia antes de baratear. Prompt caching é a alavanca #1.
