● Kinevo · Estratégia de produto

Como os MCPs entram
na Kinevo web e mobile

Hoje as 55 tools do MCP servem um cliente externo (Claude/ChatGPT). A pergunta: como trazer essa mesma inteligência para dentro do app do treinador — uma superfície de conversa que opera o Kinevo — sem duplicar catálogo nem forkar o produto.

Base: 55 tools MCP · assistente in-app já existe (3 tools) Direção: "chat-first workspace" (spec escrita) Doc: brainstorm · 15/jun/2026

1 · O que já existe (e não conversa entre si)

Em produção

MCP — 55 tools

createMcpServer(trainerId) + endpoint HTTP, auth OAuth/API-key. Cobre alunos, programas, treinos, agenda, formulários, financeiro, avaliações, insights, leads. Hints readOnly/destructive declarados.

Em produção

Assistente in-app (web)

api/assistant/chat — AI SDK, gpt-4.1-mini, streaming, só 3 tools, maxSteps:3, hardening anti-prompt-injection, rate-limit em memória. Vive no painel de comunicação.

Hoje: mensageria

Mobile (Expo)

App com abas de treinador, chat.tsx = chat treinador↔aluno (não-IA). Já consome www.kinevoapp.com via EXPO_PUBLIC_WEB_URL (oura/strava/webviews).

⚠ O problema central

Há dois cérebros desconectados: o assistente in-app sabe conversar mas só tem 3 tools; o MCP tem 55 tools mas só fala com clientes externos. O mobile não tem IA nenhuma. Trazer o MCP "pra dentro" = unificar o catálogo e plugá-lo nas superfícies certas.

2 · A chave de arco — 1 catálogo, N consumidores

A decisão técnica já fechada: ponte in-memory. O mesmo createMcpServer(trainerId) é ligado a um cliente MCP em memória (InMemoryTransport.createLinkedPair() + experimental_createMCPClient) e entregue ao streamText. Um catálogo serve todos os consumidores — sem reescrever tool nenhuma.

A mesma tool roda nos 3 caminhos. Diferença = só transporte + auth.

★ Insight desta sessão: o gate confirm já é o HITL

As tools sensíveis (create_contract, mark_payment_as_paid, cancel_contract, convert_lead) ganharam um gate confirm=true com preview/dry-run: sem confirmar, devolvem o efeito exato e não mutam nada. Isso foi feito por segurança no MCP — mas é exatamente o "human-in-the-loop" que o chat in-app precisa para ações de escrita. A UI in-app transforma esse preview em um card de confirmação (botão "Confirmar"). Mesma mecânica, dois benefícios.

3 · O que mudou desde a spec (cobertura)

4 · Entrada no WEB — superfícies

Núcleo v1

Workspace chat-first

Rota dedicada com app-shell + sidebar vivos (navegação preservada). Preferência "Chat como tela inicial" (Premium). Uma superfície, dois modos de entrada — sem forkar o dashboard.

Alto valor / baixo custo

Insights → starters acionáveis

Os assistant_insights proativos (já gerados no cron) viram botões que injetam um prompt com contexto ("Aluno X em risco → falar com ele"). Resolve a "página em branco" e absorve a dashboard.

Reuso

Command palette / ⌘K

Já existe palette. Entrada natural pro chat: "criar avaliação para Maria" abre o chat já com a intenção. Atalho onipresente sem nova tela.

Para as lacunas

Deep-link pra GUI

Treino ao vivo, settings e fluxos sem tool → o chat leva o treinador pra tela certa em vez de fingir que executa. Honesto e barato; nenhuma tool nova no v1.

Decisão de implementação (já investigada)

AI SDK 4.3.19 descarta as annotations das tools MCP (lê só name/description/schema) e dá execute automático a todas. Logo: (1) a classificação read/write tem que ser estática em tool-policy.ts; (2) HITL exige tirar o execute das tools de confirmação e ter um endpoint execute-tool que reusa a ponte. Imports validados.

5 · Entrada no MOBILE — o caminho mais curto

O mobile não precisa de cérebro próprio: ele consome a mesma /api/assistant/chat que o web. A ponte e as tools vivem no servidor Next; o app só renderiza a conversa. Auth = token da sessão Supabase que o app já tem.

Reaproveita backend

Tela de chat IA nativa

Nova aba/tela RN que streama de www.kinevoapp.com/api/assistant/chat com o JWT do Supabase. Render nativo das mensagens + dos cards de confirmação (HITL) e de deep-link.

Vantagem mobile

Voz → ação

"Marca o pagamento da Maria como pago" por voz (speech-to-text nativo) → preview → confirma com Face ID. O mobile é onde o treinador está na quadra/academia, sem teclado.

Push como gancho

Insight → push → chat

O cron já manda push de insight. Tocar no push abre o chat já com o starter daquele aluno. Loop proativo nativo: notificação que vira ação em 2 toques.

Cuidados do mobile

Deep-link reverso: quando o chat não tem tool (treino ao vivo), precisa navegar pra rota Expo certa — mapear tool-gaps → rotas app/*. Streaming: validar SSE/streaming no fetch RN (ou cair pra polling). Custo: voz incentiva uso pesado → o metering por créditos vale ainda mais aqui. Deploy: tela nova = release na App Store/Play (web é instantâneo) — começar pelo web, mobile como fast-follow.

6 · Transversais (valem pros dois)

TemaDecisão / estado
AuthIn-app (web+mobile) = sessão Supabase (sem OAuth). O trainerId sai da sessão e entra na ponte. Externo segue OAuth.
HITL / escritaReusar o gate confirm das tools sensíveis como card de confirmação. Tools destructive nunca auto-executam.
MeteringRate-limit atual é em memória (anti-abuso, não billing). Cota real precisa de tabela (ai_usage_periods/ai_usage_events + RPC atômica). Medir por créditos (prescrição pesa ~10×), capturar usage no onFinish.
Planos / tierHoje acesso é binário (active/trialing). Precisa trainers.ai_tier + popular subscriptions.stripe_price_id. Tiers: Essencial R$79,90 · Pro IA R$129,90 · Premium IA R$199,90.
Custo / modeloChat é input-dominated (system prompt c/ 55 schemas re-enviado por step). Prompt caching é a alavanca #1. Rotear "gera programa" pra action determinística (1 call) em vez de 20 writes.
SegurançaHardening anti-prompt-injection já existe no chat. Tools são isoladas por trainerId (admin client + ownership explícito). Não reduzir isso.

⚠ Custo escala com sucesso

Power-user via chat ≈ R$180/mês de API (prescrição-via-chat ~R$0,55/turno, 15-25 tool calls). O cap de créditos é proteção de margem, não só UX — tem que existir antes de abrir a feature.

7 · Sequência sugerida

F0

Fundações

Ponte in-memory + tool-policy.ts (read/write/peso) + metering (tabela + RPC) + trainers.ai_tier. Nada visível ainda. prompt F0 já existe na spec

F1

Workspace MVP (web)

Rota de chat com as 55 tools via ponte + HITL (cards de confirmação reusando o gate confirm) + gate de plano. Deep-link pras lacunas.

F2

Chat-as-home + insights-starters (web)

Preferência "chat como tela inicial" (Premium) + insights proativos viram botões de conversa. Aqui o chat "absorve" a dashboard.

F3

Mobile fast-follow

Tela de chat IA nativa consumindo a mesma API + voz + push→chat. Mapear deep-links reversos pras rotas Expo. Release App Store/Play.

8 · Decisões em aberto pra discutir

1
Web-first ou paralelo? Mobile reusa 90% do backend, mas tela nova = release nas lojas. Sugiro web F1→F2, mobile como F3 — concorda?
2
Quão "chat-first"? Chat absorve a dashboard (tela inicial Premium) ou fica como painel/aba ao lado da GUI atual? A spec aponta pro primeiro, faseado.
3
Escopo de escrita no v1. Liberar todas as 55 (incl. financeiro/cancelar com confirm) ou começar só leitura + comunicação + prescrição, e abrir o resto por fase?
4
Modelo de cobrança. Créditos por plano (Essencial/Pro/Premium) com reset semanal + top-up — fechar pesos e caps antes de codar.
5
Voz no mobile entra no v1 mobile ou é um v2? Muda a expectativa de custo/uso.
